# Decisões Técnicas (ADRs)

Este documento registra as principais Decisões de Arquitetura (Architecture Decision Records) do projeto, explicando o "porquê" de cada escolha e seus trade-offs.

---

## ADR-001: Usar Mintlify MDX para Documentação

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Documentação em Mintlify (MDX) vs. alternativas |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Necessidade de documentação navegável, responsiva e fácil de manter para avaliação do desafio técnico.

### Alternativas Consideradas

1. **Docusaurus** — Similar ao Mintlify, comunidade grande, mas menos elegante
2. **Nextra** — Ótimo para docs, mas menos visual
3. **Readme.md único** — Simples, mas não atende ao requisito de "documentação robusta" navegável

### Decisão

**Usar Mintlify com páginas MDX** (pedido explícito do usuário)

### Justificativa

- ✅ Tema responsivo nativo (mobile-first)
- ✅ Busca integrada e SEO automático
- ✅ Suporte a componentes ricos (Cards, Tabs, Callouts)
- ✅ Versionamento no Git (documentação como código)
- ✅ Deploy trivial via `mint dev` em container

### Trade-offs

- **Con**: Menos popular que Docusaurus em Brasil
- **Con**: Plano Enterprise necessário para self-hosted oficial (mitigado com `mint dev` em container)
- **Pro**: Setup mais rápido, menos configuração

### Impacto

Documentação de alta qualidade, avaliador consegue entender projeto em menos de 10 minutos.

## ADR-002: Node.js 20 LTS para Backend e Documentação

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Node.js 20 LTS (≥20.17, menores que 25) |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Necessidade de padronizar versão de Node para frontend (Next.js), backend (Fastify) e documentação (Mintlify CLI).

### Alternativas Consideradas

1. **Node 18 LTS** — Older, suporte termina em 2024
2. **Node 22 LTS** — Futuro, mas ainda em early stage
3. **Node 20 LTS** — Maduro, suporte até 2026

### Decisão

**Node.js 20 LTS** (>=20.17, menores que 25)

### Justificativa

- ✅ LTS estável, suporte até 2026-04
- ✅ Suporte total a ESM (ECMAScript Modules)
- ✅ TypeScript 5.x compatível
- ✅ Mint CLI (>=20.17) requerido — exigência do Mintlify
- ✅ Imagens Docker `node:20-alpine` são pequenas (40MB)

### Trade-offs

- **Con**: Não suporta Node 25+ (futuro, pode requerer upgrade)
- **Pro**: Paridade perfeita entre local e produção (Railway)

### Impacto

Ambiente consistente, builds reproduzíveis, sem surpresas entre dev e prod.

## ADR-003: Groq Cloud para IA (vs OpenAI/Anthropic)

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Groq Cloud (GPT-OSS-120B) para chat e geração de guia |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Necessidade de integração com LLM para:
1. Geração de guia de experiências (restaurantes, atrações)
2. Chat streaming do assistente virtual

### Alternativas Consideradas

| Critério | Groq | OpenAI | Anthropic |
|----------|------|--------|-----------|
| **Latência** | ~400ms ⭐ | ~1000ms | ~1200ms |
| **Throughput (tok/s)** | 150+ ⭐ | 80-100 | 100 |
| **Custo ($/M tokens)** | $0.05 ⭐ | $0.10 | $0.15 |
| **Modelo** | GPT-OSS-120B | GPT-4 | Claude 3.5 |
| **Rate Limit** | Alto | Médio | Médio |

### Decisão

**Groq Cloud para tarefas de latência crítica** (chat streaming)

### Justificativa

- ✅ **Velocidade**: 150+ tok/s vs 80-100 (OpenAI) — respostas aparecem 2x mais rápido
- ✅ **Custo**: 50% do custo de OpenAI — importante em 48h de avaliação
- ✅ **Modelo**: GPT-OSS-120B é competente para tarefas simples (restaurantes, rules)
- ✅ **Vercel AI SDK**: Provider-agnostic, trocar de Groq para OpenAI é trivial

### Alternativas Rejeitadas

- **OpenAI**: Mais caro, mais lento, desnecessário para tarefas simples
- **Anthropic Claude**: Excelente para raciocínio complexo, mas não necessário aqui

### Trade-offs

- **Con**: Groq é mais nova (fundada 2023), menos historia em prod
- **Con**: Se qualidade cair, fallback para OpenAI requer apenas 2 linhas de código
- **Pro**: Streaming é **muito** mais rápido, melhor UX

### Fallback

Se Groq se mostra instável:
```typescript
// Trocar provider em runtime
const provider = process.env.AI_PROVIDER === 'openai' 
  ? 'openai'
  : 'groq';
```

### Impacto

Chat responsivo (tokens aparecem em 300ms), custo 50% menor, melhor UX geral.

## ADR-004: Persistência de Guia de Experiências (Cache 1:1)

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Gerar guia uma vez, reutilizar em acessos posteriores |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Geração de guia por IA leva 2-3 segundos (rede + processamento). Hóspede acessa guia múltiplas vezes. Regenerar a cada acesso é:
- ❌ Lento (esperar 2-3s sempre)
- ❌ Caro (pagar por tokens novamente)
- ❌ Inconsistente (recomendações mudam a cada reload)

### Alternativas Consideradas

1. **Gerar sempre** — Simples, mas lento e caro
2. **Cache em Redis** — Rápido, mas complexidade extra
3. **Cache no BD (Relação 1:1)** — Simples, rápido, sem dependências

### Decisão

**Tabela `Guidebook` (1:1 com `Property`)**

```prisma
model Property {
  id Int @id
  code String @unique
  // ... 20+ campos
  guidebook Guidebook? // Relação 1:1
}

model Guidebook {
  id Int @id
  propertyId Int @unique // Foreign key
  restaurants Json
  attractions Json
  essentials Json
  seasonal String
  createdAt DateTime
}
```

### Justificativa

- ✅ **Primeira vez**: Query `guidebook IS NULL` → gerar com IA
- ✅ **Próximas vezes**: Query `guidebook IS NOT NULL` → retornar cache (menos de 100ms)
- ✅ **Consistência**: Mesmo guia sempre (não muda com IA updates)
- ✅ **Simplicidade**: Sem Redis, sem cache invalidation complexa

### Trade-offs

- **Con**: Guia fica "congelado" (novo tipo de restaurante não aparece automaticamente)
- **Pro**: Compensado por `EXPIRY_DAYS = 30` (regenerar se >30 dias)

### Implementação

```typescript
// 1. Verificar se existe
const existing = await prisma.guidebook.findUnique({
  where: { propertyId }
});

if (existing) return existing; // Cache hit

// 2. Gerar novo
const guidebook = await generateWithAI(property);

// 3. Persistir
await prisma.guidebook.create({
  data: { propertyId, ...guidebook }
});
```

### Impacto

Performance 20x melhor (2-3s → 100ms), custo 80% menor, UX melhor.

## ADR-005: Vertical Slices (vs Camadas Tradicionais)

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Organizar por feature (slices) vs por tipo de arquivo (camadas) |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Estrutura de código afeta:
- Velocidade de desenvolvimento
- Facilidade de testes
- Capacidade de paralelizar trabalho
- Compreensão de features

### Alternativas Consideradas

1. **Por Camadas** (tradicional)
   ```
   controllers/  (100+ arquivos)
   services/     (50+ arquivos)
   models/       (30+ arquivos)
   ```

2. **Por Feature (Vertical Slices)** (proposto)
   ```
   slices/
     property/     (tudo relacionado a imóvel)
     guidebook/    (tudo relacionado a guia)
     chat/         (tudo relacionado a chat)
   ```

### Decisão

**Vertical Slices (Feature-Based)**

### Justificativa

- ✅ **Paralelização**: Dev 1 trabalha em `property/`, Dev 2 em `guidebook/` — zero conflitos
- ✅ **Compreensão**: Novo dev lê apenas `slices/chat/` para entender chat completo
- ✅ **Testes**: Testes da feature vivem perto do código
- ✅ **SOLID**: Single Responsibility, Open/Closed naturais

### Alternativa Rejeitada

Camadas tradicionais levam a:
- ❌ Acoplamento inadvertido entre features
- ❌ Pasta `components/` cresce infinitamente
- ❌ Difícil fazer feature-branch isolada
- ❌ Não paraleliza bem

### Impacto

Código mais mantenível, testes independentes, desenvolvimento mais rápido, menos bugs de integração.

## ADR-006: Streaming SSE para Chat (vs WebSocket/Polling)

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | SSE (Server-Sent Events) para chat streaming |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Chat precisa mostrar resposta da IA progressivamente. Opções:
1. **Polling** — Cliente pergunta a cada 100ms "chegou resposta?"
2. **WebSocket** — Conexão bidirecional persistente
3. **SSE** — Unidirecional, server envia eventos para cliente

### Alternativas Consideradas

| Critério | Polling | WebSocket | SSE |
|----------|---------|-----------|-----|
| **Latência** | 100-500ms | 10ms | 10ms |
| **Banda** | Alta ❌ | Média | Baixa ✅ |
| **Setup** | Trivial ✅ | Complexo | Trivial ✅ |
| **Navegadores** | Todos | Todos | Todos |
| **Fallback** | N/A | HTTP long-polling | N/A |

### Decisão

**SSE (Server-Sent Events)**

### Justificativa

- ✅ **Simples**: Apenas `response.stream` no backend, `eventSource` no frontend
- ✅ **Rápido**: Latência tão baixa quanto WebSocket
- ✅ **Eficiente**: Unidirecional (client→server é POST, server→client é SSE)
- ✅ **Suporte**: Todos os navegadores modernos
- ✅ **Fastify**: Suporte nativo com `Reply.raw.write()`

### Implementação

**Backend (Fastify)**:
```typescript
reply.type('text/event-stream');
reply.header('Connection', 'keep-alive');

for await (const chunk of stream) {
  reply.raw.write(`data: ${JSON.stringify(chunk)}\n\n`);
}
```

**Frontend (React)**:
```typescript
const eventSource = new EventSource('/api/chat?message=...');
eventSource.onmessage = (e) => {
  const { token } = JSON.parse(e.data);
  setResponse(prev => prev + token);
};
```

### Trade-offs

- **Con**: SSE é unidirecional (cliente não pode interromper server direto)
- **Pro**: Mitigado com `AbortController` (cliente fecha `fetch`)

### Impacto

Chat responsivo, tokens aparecem em tempo real, UX similar a ChatGPT.

## ADR-007: Zod para Validação de Fronteira

| Aspecto | Detalhes |
|---------|----------|
| **Decisão** | Zod 3.x para schema validation |
| **Data** | 2026-06-19 |
| **Status** | ACEITO |

### Contexto

Validação de dados:
- Input do hóspede (código de imóvel, mensagem de chat)
- Response da IA (restaurantes, atrações)
- Dados do banco (Property, Guidebook)

### Alternativas Consideradas

1. **Sem validação** — Rápido, mas bugs silenciosos
2. **Validação manual** (`typeof`, `if`)— Verboso, propenso a erros
3. **Zod** — Type-safe, DRY, gera tipos automaticamente

### Decisão

**Zod 3.x para todas as fronteiras (input/output)**

### Justificativa

- ✅ **Type Safety**: `z.infer<typeof schema>` gera tipos automaticamente
- ✅ **DRY**: Uma definição → validação + tipos
- ✅ **Zero Dependências**: Não adiciona bibliotecas extras
- ✅ **Erros Claros**: Mensagens de erro úteis para API

### Exemplo

```typescript
// Uma definição...
const propertyCodeSchema = z.string()
  .min(3).max(10)
  .regex(/^[A-Z0-9]+$/);

// Gera automaticamente:
// - Validação em runtime
// - Tipo TypeScript
// - Documentação de erro
type PropertyCode = z.infer<typeof propertyCodeSchema>;

const result = propertyCodeSchema.parse(userInput);
// → Error: "Código deve ter 3-10 caracteres"
```

### Impacto

Menos bugs de type mismatch, melhor UX (erros claros), código mais mantenível.
